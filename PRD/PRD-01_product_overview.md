## 0. 이 문서의 역할과 사용법

이 문서는 **"무엇을 왜 만드는가"** 에 대한 단일 진실 원천(Single Source of Truth)이다.  
구현 중 결정이 필요한 모든 순간, 이 문서의 **3.1 비즈니스 KPI** 와 **2.2 핵심 가치 제안**이 판단 기준이 된다.

### 읽는 순서
1. §1 배경 → §2 제품 정의 → §3 성공 지표 순서로 읽는다
2. MVP 범위가 불명확할 때는 §2.3 In/Out Scope 표를 참조한다
3. 기능 우선순위 결정이 필요할 때는 §3.2 품질 KPI를 기준으로 한다

## 1. 배경 및 문제 정의

### 1.1 타겟 시장

| 항목 | 내용 |
|------|------|
| **주요 타겟** | 자동차 부품 제조 중견·중소기업 기획팀 담당자 |
| **기업 규모** | 상시 직원 300~2,000명, 연 매출 500억~5,000억 원 |
| **지역** | 경남·경북·충남·울산 등 자동차 부품 제조 클러스터 |
| **담당 업무** | 정부 R&D·스마트공장·DX 지원사업 발굴 및 신청 |
| **연간 신청 건수** | 담당자 1인당 3~5건 신청, 10~20건 공고 검토 |

### 1.2 핵심 페인포인트 (정량 데이터)

| # | 문제 | 현재 해결 방법 | 측정된 비용 |
|---|------|---------------|------------|
| P1 | 정부지원 공고가 기업마당·중기부·과기부·창진원·지자체 등 **12개 이상 채널**에 분산 | 담당자가 주 1~2회 각 사이트 직접 방문·북마크 관리 | **주 3~5시간** (연 156~260시간) |
| P2 | 공고 PDF 첨부파일(평균 15~40페이지) 수작업 해석 | 직접 정독 후 Excel 정리 | **건당 1~2시간** |
| P3 | 우리 회사 조건(업종·규모·지역·기구축 수준) 적합 여부 판단 어려움 | 담당자 경험에 의존, 선배·동료에게 물어봄 | 적합 공고 **30% 누락** 추정 |
| P4 | 사업계획서·신청서 매번 신규 작성 | 이전 신청서 복사·붙여넣기 후 수정 | **건당 8~20시간** (연 24~100시간) |
| P5 | 신청 후 탈락 원인 불명 | 사후 분석 없음 | 다음 신청 개선 불가, 기회비용 손실 |
| P6 | 동시수행 제한 조건 파악 누락 | 신청 후 자격 미달로 반려 | 신청 준비 기간 **2~4주 낭비** |

**연간 총 낭비 비용 추산** (담당자 1인 기준):
- 공고 탐색: 200시간 × 30,000원/h = **600만 원**
- 사업계획서 작성: 60시간 × 30,000원/h = **180만 원**
- **합계: 연 약 780만 원/인** (기회비용 제외)

### 1.3 현재 해결책의 한계

**방법 1: 수작업 직접 탐색**
- 한계: 채널이 너무 많아 누락 필연적, 확장 불가

**방법 2: ChatGPT/Claude에 직접 공고문 붙여넣기**
| 기능 | ChatGPT 직접 | AJIN BizAI |
|------|-------------|-----------|
| 공고 자동 수집 | ❌ 없음 | ✅ 4개 API 자동 |
| 기업 프로필 기억 | ❌ 세션마다 초기화 | ✅ localStorage 영구 저장 |
| 적합도 수치화 | ❌ 주관적 판단 | ✅ AI 유사도 % 제시 |
| 평가기준 파싱 | ❌ 수동 붙여넣기 필요 | ✅ 공고문 자동 분석 |
| 부족정보 진단 | ❌ 없음 | ✅ 역방향 인터뷰 UX |
| 선정 점수 시뮬레이션 | ❌ 없음 | ✅ 항목별 예상 점수 |
| 신청 이력 관리 | ❌ 없음 | ✅ 동시수행 제한 추적 |
| 한국 공고 특화 | ❌ 범용 | ✅ 스마트공장/R&D 특화 |

**방법 3: 전문 컨설팅 업체 활용**
- 한계: 건당 100~500만 원 비용, 신청 시즌에 대응 지연

## 2. 제품 정의

### 2.1 제품 한 줄 정의 (Elevator Pitch)

> **AJIN BizAI는 자동차 부품 제조기업 기획팀이 정부지원사업 공고를 자동 수집·추천받고, AI와 대화하며 사업계획서 초안을 작성하고, 선정 가능성을 사전에 시뮬레이션할 수 있는 원스탑 AI 플랫폼이다.**

### 2.2 핵심 가치 제안 (3가지 차별점)

#### 차별점 1: 역방향 데이터 수집 UX
- 기존: "기업 정보를 업로드하세요" → 사용자가 무엇을 줘야 할지 모름
- AJIN: "이 공고를 쓰려면 불량률 수치가 필요한데, 현재 수치가 어떻게 되나요?" → AI가 먼저 물어봄
- 결과: 작성 가능률 64% → 92% 상승 (사용자 인터뷰 기반 추정)

#### 차별점 2: 선정 점수 시뮬레이션
- 완성된 초안을 공고의 평가지표로 역분석
- "현재 67점 / 커트라인 추정 75점, 이 항목 수정 시 +8점"
- 공고문을 완전히 이해한 플랫폼만 가능한 기능

#### 차별점 3: 신청 이력 동시수행 추적
- 현재 수행 중인 사업 자동 추적
- 신규 공고 진입 시 동시수행 제한 자동 경고
- 담당자 교체 시에도 이력 유지

### 2.3 MVP 범위 (v1.0 = Day 1~56)

| 모듈 | 포함 | 제외 이유 |
|------|------|----------|
| 공고 수집 (4개 API) | ✅ | - |

| 자료 업로드 + AI 진단 | ✅ | - |
| AI 인터뷰형 데이터 보완 | ✅ | - |
| 사업계획서 AI 초안 생성 (2모드) | ✅ | - |
| 선정 평가 점수 시뮬레이션 | ✅ | - |
| 히스토리 + 동시수행 추적 | ✅ | - |

## 3. 성공 지표 (KPI)

### 3.1 비즈니스 KPI (v1.0 목표, Day 56 기준)

| KPI | 목표값 | 측정 방법 |
|-----|--------|---------|
| 공고 수집 건수 | 1,000건 이상 동시 관리 | 필터 결과 카운터 |
| 적합 공고 추천 정확도 | 담당자 평가 80% 이상 "적합" | 사용자 피드백 |
| 사업계획서 초안 생성 성공률 | 95% 이상 (LLM 오류 제외) | 에러 로그 |
| 초안 작성 시간 단축 | 기존 대비 70% 감소 (20h → 6h) | 사용자 설문 |

### 3.2 제품 품질 KPI (구현 완료 기준)

| KPI | 목표값 | 측정 도구 |
|-----|--------|---------|
| 초기 로딩 LCP | < 3초 | Chrome Lighthouse |
| 공고 목록 로딩 | < 5초 (4개 API 병렬) | `performance.now()` |
| AI 적합도 계산 | < 2초 (1,000건 기준) | `console.time()` |
| LLM 초안 생성 | 섹션당 < 30초 | 타임아웃 30,000ms |
| PDF 파싱 | < 10초 (50MB) | 백엔드 응답 시간 |
| 단위 테스트 커버리지 | 80% 이상 | Jest coverage |
| 접근성 점수 | Lighthouse 90 이상 | Lighthouse |

### 3.3 사용자 경험 KPI

| KPI | 목표값 |
|-----|--------|
| 공고 검색 → 초안 생성 완료 소요 시간 | < 30분 |
| 작성 가능률 초기값 | 60~70% (업로드 직후) |
| 작성 가능률 AI 보완 후 | 85~95% |
| 오류 발생 시 사용자가 대응 가능한 비율 | 100% (모든 에러 한국어 안내) |

## 4. 제약 조건

### 4.1 기술 제약

| 제약 | 내용 | 대응 방법 |
|------|------|---------|
| 한국 공공 API | 일일 호출 한도 존재 (기관별 상이), CORS 제한 | Express 프록시 서버로 우회, 캐시(1시간) |
| HWP 파일 파싱 | Python 라이브러리 불완전 (`python-hwplib` 비공식) | 파싱 실패 시 텍스트 직접 입력 안내 |
| LM Studio 로컬 실행 | 팀 서버 없을 경우 각 PC에서 실행 필요 | OpenAI/Anthropic API 키로 전환 가능하게 설계 |
| Sentence Transformer 모델 크기 | `jhgan/ko-sroberta-multitask` = 약 500MB | 초기 로딩 1회, 이후 메모리 캐시 |
| PDF 미리보기 | iframe CORS 제한으로 외부 PDF 직접 임베드 불가 | 다운로드 링크 제공 + 텍스트 추출 결과 표시 |

### 4.2 법적 제약

| 항목 | 제약 | 준수 방법 |
|------|------|---------|
| 공공 API 이용 약관 | 상업적 재판매 금지, 출처 명시 의무 | 공고 카드에 출처 기관명 뱃지 표시 |
| 개인정보보호법 | 기업 재무정보 등 민감 정보 처리 | 서버 미전송, localStorage만 사용 (MVP) |
| 저작권 | 공고 원문 복제 제한 | 요약만 제공, 원문 링크로 연결 |

### 4.3 UX 제약

- **비개발자 대상**: 기획팀 담당자는 CLI, 개발 환경 설정 불가 → 모든 설정은 웹 UI에서 가능해야 함
- **HWP 의존성**: 한국 기업 문서 90% 이상이 HWP → HWP 업로드 반드시 지원 (파싱 실패해도 업로드 자체는 가능)
- **인터넷 환경**: 사내 방화벽으로 일부 외부 API 차단 가능 → Mock 폴백 필수
